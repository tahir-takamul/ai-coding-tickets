# T03 — Port `android-build` module to mobile-automated + wire into bootstrap-orchestrator

**Status**: pending (orchestrator wired with stub; module logic pending port)
**Repo**: `takamulai/mithril` — `iac/cicd-onprem/pipelines/mobile-automated/modules/android-build.Jenkinsfile`
**Blocks**: T05 (upload needs a built artefact)
**Blocked by**: T01, T02

## Reason

This is the build step that produces the bootstrap-flavour APK. The legacy
module at `iac/cicd-onprem/pipelines/mobile/modules/android-build.Jenkinsfile`
already implements the full recipe end-to-end (workspace hygiene,
checkout, tool verification, signing-credential injection, gradle
`assembleBootstrapRelease`, archive). Port verbatim with mobile-automated path
tweaks; behaviour stays identical.

The relevant Jenkins artefact-handoff behaviour is documented in
[[FINDING §F01]], [[FINDING §F02]], [[FINDING §F03]] — read these before
making *any* change to the archive / cleanup logic.

## Input

- Read `iac/cicd-onprem/pipelines/mobile/modules/android-build.Jenkinsfile`
  — the source module to port.
- Confirm credential IDs follow the `rcbdc-<thing>-<environment>`
  convention (the legacy header documents these); the
  bootstrap-orchestrator passes `ENVIRONMENT=bootstrap`.
- Confirm `cbdc_retail` agent label is correct for mobile-automated builds.

## Output

### `iac/cicd-onprem/pipelines/mobile-automated/modules/android-build.Jenkinsfile`

Port the legacy module with these changes:

- Header references `mobile-automated/` and links to this T03 spec.
- `BRANCH` default → `staging` (was `main` in legacy).
- `BITBUCKET_REPO_URL` unchanged — same repo. Both pipelines build
  from Bitbucket; mobile-automated just uses the `staging` branch.
- `COMMIT_SHA` parameter retained — the orchestrator passes
  `RESOLVED_SHA` (from T01) so the build is pinned to the exact ref
  that was sync'd.
- `BUILD_TAG` formula updated to include the resolved version:
  `"${RESOLVED_VERSION_LABEL}-${BUILD_NUMBER}-${shortSha}"`. New
  optional parameter `RESOLVED_VERSION_LABEL` (default `staging`) — the
  orchestrator passes `RESOLVED_VERSION` (a tag name like `v1.2.3`, or
  the sentinel `staging-HEAD`).
- All credential resolution unchanged (`rcbdc-wallet-env-bootstrap`,
  `rcbdc-google-services-bootstrap`, `rcbdc-wallet-keystore-bootstrap`,
  `rcbdc-cbuae-root-ca`).
- `archiveArtifacts` + `cleanWs` retention unchanged — see [[FINDING §F03]].

### Bootstrap orchestrator wiring (already done; this spec documents contract)

`mobile-automated/orchestrators/bootstrap-orchestrator.Jenkinsfile`:

- `Build Android` stage gated by `when { expression { params.OS in ['Android','Both'] } }`.
- Calls `build job: 'Retail-CBDC/00-Mobile/Modules/android-build'` with
  `BRANCH=staging`, `COMMIT_SHA=${env.RESOLVED_SHA}`,
  `ENVIRONMENT=${env.TARGET_ENV}` (bootstrap),
  `RESOLVED_VERSION_LABEL=${env.RESOLVED_VERSION}`.
- Captures `result.buildVariables.BUILD_TAG`, `result.fullProjectName`,
  `result.number` into env vars for the upload stage (see [[FINDING §F04]]).

## Acceptance criteria

- `mobile-automated/modules/android-build.Jenkinsfile` exists and builds an APK
  identical to what the legacy module produces (same gradle task, same
  signing config, same archive path).
- For a tag-build (`VERSION_TAG=v1.2.3`), `BUILD_TAG` looks like
  `v1.2.3-42-a1b2c3d`.
- For a HEAD-build (`VERSION_TAG=""`), `BUILD_TAG` looks like
  `staging-HEAD-42-a1b2c3d`.
- The APK survives the cleanWs hook and is reachable from
  `$JENKINS_HOME/jobs/Retail-CBDC/jobs/00-Mobile/jobs/Modules/jobs/android-build/builds/<N>/archive/...`
  (see [[FINDING §F04]] for path translation).

## Notes

- Do not skip `archiveArtifacts` — it is the only thing that lets the
  upload module find the file later. See [[FINDING §F03]].
- Workspace cache (`node_modules`, `.gradle`, `build`) preserved
  between runs — large speedup. Don't toggle without checking with
  the mobile team.
