# T01 — Initialise `mobile-v2/` + bootstrap-orchestrator sync step

**Status**: DONE
**Repo**: `takamulai/mithril` — `iac/cicd-onprem/pipelines/mobile-v2/`
**Blocks**: T02 (next-step stages, to be defined)
**Blocked by**: —

## Reason

The existing `mobile/` orchestrator-based pipeline is locked for the
current release. Mobile-v2 is a greenfield rebuild that starts from the
bootstrap-orchestrator and grows downward. The very first stage must be a
GitHub→Bitbucket sync, because Jenkins on-prem only has SCM access to
Bitbucket (`dsocodemgmt.cbuae.gov`), while developers push to GitHub
(`takamulai/mithril`).

The legacy module at
`iac/cicd-onprem/pipelines/modules/sync-github-to-bitbucket.Jenkinsfile`
uses `--no-tags` to skip the noisy build-pipeline tags (iOS TestFlight,
etc.). For mobile-v2, the release model is "every commit on `staging` is
tagged with a version, and the bootstrap-orchestrator builds a specific tag" — so we **must**
mirror staging-reachable tags too, and the orchestrator must be able to
resolve a version tag.

## Input

- Read `iac/cicd-onprem/pipelines/modules/sync-github-to-bitbucket.Jenkinsfile`

- Read `iac/cicd-onprem/pipelines/mobile/orchestrators/sit-orchestrator.Jenkinsfile`

- Confirm GitHub URL `https://github.com/takamulai/mithril.git` and

## Output

### `iac/cicd-onprem/pipelines/mobile-v2/` (new tree)

```
mobile-v2/
├── modules/
│   └── sync-github-to-bitbucket.Jenkinsfile
└── orchestrators/

```

### `mobile-v2/modules/sync-github-to-bitbucket.Jenkinsfile`

- Mirrors legacy shape (agent `cbdc_retail`, `cbdc-bb-creds`,
  `--force-with-lease` push) **with these deltas**:
  - Hard-coded source: GitHub branch `staging` → Bitbucket branch `staging`
    (created on first run; Bitbucket's default branch `main` is untouched
    and continues to serve the legacy `mobile/` pipeline). No
    `GITHUB_BRANCH` parameter (mobile-v2 has a fixed mapping).
  - Fetch all GitHub tags after the branch fetch, then compute the
    set of tags whose commits are reachable from `github/staging`, and
    push them one-by-one to Bitbucket using
    `refs/tags/<tag>:refs/tags/<tag>`. This avoids dragging the
    TestFlight / feature-branch tags that aren't on staging.
  - Idempotent: if both branch and all relevant tags are already in
    sync, the stage exits cleanly with no force-push.

### `mobile-v2/orchestrators/bootstrap-orchestrator.Jenkinsfile`

Parameters:
- `VERSION_TAG` (string, default empty) — desired release tag. Empty
  ⇒ check out Bitbucket `staging` HEAD as-is (no tag lookup).

Stages:
1. **Init** — print params, initialise tracking vars
   (`SYNC_STATUS`, `RESOLVED_VERSION`, …) used by `post { always }`.
2. **Sync GitHub → Bitbucket** — `build job:` into the new sync module,
   `propagate: false`, capture result.
3. **Resolve Version** — if `VERSION_TAG` is provided, verifies the
   tag exists on Bitbucket and checks it out; otherwise checks out the
   HEAD of `bitbucket/staging`. Sets `env.RESOLVED_VERSION` (the tag
   name, or the sentinel `staging-HEAD`) and `env.RESOLVED_SHA`.

> Subsequent stages (security scans, builds, uploads) are intentionally
> **not** written in T01. They are defined in T02+ once the user
> confirms the next step.

## Acceptance criteria

- `mobile-v2/` directory exists alongside `mobile/`; legacy `mobile/`
  tree is untouched.
- New sync module pushes GitHub `staging` → Bitbucket `staging` and
  tags reachable from `staging`; idempotent on re-run with no diff;
  Bitbucket's `main` branch is untouched.
- Bootstrap orchestrator runs Init → Sync → Resolve Version; with
  `VERSION_TAG=""` it checks out Bitbucket `staging` HEAD and sets
  `RESOLVED_VERSION=staging-HEAD`; with `VERSION_TAG=v1.2.3` it checks
  that tag out on Bitbucket `staging`.
- `post { always }` prints sync status, resolved version, overall
  result.

## Notes

- `agent { label 'cbdc_retail' }` mirrors the legacy sync agent; the
  orchestrator stays on `Jenkins` (lightweight, only calls modules).
- Sync still uses `--force-with-lease` — protects against silently
  wiping a Bitbucket-side hotfix on `staging`. On the first run, `staging`
  doesn't yet exist on Bitbucket and the push creates it.
- We deliberately push tags **one-by-one** (`refs/tags/x:refs/tags/x`)
  rather than `git push --tags`, so we can scope to the staging-reachable
  set and surface per-tag failures clearly in the log.
- Job paths in the orchestrator default to
  `Retail-CBDC/00-Mobile/Modules/sync-github-to-bitbucket` — the
  Jenkins folder convention isn't locked yet; this is the assumed name
  and is easy to flip if the platform team picks a different folder.
