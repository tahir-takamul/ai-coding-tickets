# T02 — Port `security-scan` module to mobile2 + wire into bootstrap-orchestrator

**Status**: pending (orchestrator wired with stub; module logic pending port)
**Repo**: `takamulai/mithril` — `iac/cicd-onprem/pipelines/mobile2/modules/security-scan.Jenkinsfile`
**Blocks**: T03 (security scans run before build)
**Blocked by**: T01

## Reason

Mobile2's bootstrap-orchestrator must run SonarQube + Checkmarx SAST + Checkmarx SCA before
any build artefact is produced (analysing source code, not built bits,
catches issues cheaper). The legacy module
`iac/cicd-onprem/pipelines/mobile/modules/security-scan.Jenkinsfile`
already implements this for all three scan types via a single
`SCAN_TYPE` parameter — port verbatim, with mobile2-specific path
defaults.

Mobile2 also adds a **`ENABLE_SECURITY_SCANS` boolean** (default true)
to the bootstrap-orchestrator so the operator can skip scans for hotfix /
debugging runs.

## Input

- Read `iac/cicd-onprem/pipelines/mobile/modules/security-scan.Jenkinsfile`
  — the source module to port.
- Read `iac/cicd-onprem/pipelines/mobile/orchestrators/sit-orchestrator.Jenkinsfile`
  lines 94-172 — the existing parallel-scans block, to mirror in mobile2.
- Confirm SonarQube/Checkmarx server URLs and credential IDs from the
  legacy module — reused as-is.

## Output

### `iac/cicd-onprem/pipelines/mobile2/modules/security-scan.Jenkinsfile`

Port the legacy module with these changes:

- Header comment references `mobile2/` and links to this T02 spec.
- `BRANCH` default → `staging` (not `main`); mobile2 builds from
  Bitbucket `staging`.
- Adds optional `COMMIT_SHA` parameter so the orchestrator can pin the
  scan to the resolved version's SHA (see T01's `RESOLVED_SHA`).
- Same `SCAN_TYPE` choice param (`sonarqube`, `checkmarx-sast`,
  `checkmarx-sca`).
- Same `FAIL_ON_VIOLATION` boolean (legacy default `false`).
- Same `ENVIRONMENT` choice param (`bootstrap`, `uat`, `preprod`,
  `prod`).

### Bootstrap orchestrator wiring (already done; this spec
documents the contract)

`mobile2/orchestrators/bootstrap-orchestrator.Jenkinsfile`:

- New parameter `ENABLE_SECURITY_SCANS` — `booleanParam`, default `true`.
- `Security Scans` parallel stage gated by `when { expression { params.ENABLE_SECURITY_SCANS } }`.
- Each sub-stage calls `build job: 'Retail-CBDC/00-Mobile/Modules/security-scan'`
  with the appropriate `SCAN_TYPE` and the resolved `BRANCH=staging`,
  `COMMIT_SHA=${env.RESOLVED_SHA}`.

## Acceptance criteria

- `mobile2/modules/security-scan.Jenkinsfile` exists and runs the three
  scan types via `SCAN_TYPE` parameter, matching legacy behaviour.
- With `ENABLE_SECURITY_SCANS=true` (default) the bootstrap-orchestrator runs
  all three scans in parallel.
- With `ENABLE_SECURITY_SCANS=false` the `Security Scans` stage shows
  as `Skipped` in the Jenkins UI and the bootstrap pipeline proceeds
  straight to the Build stage.
- A scan failure (Sonar quality gate red, Checkmarx high severity, …)
  marks the relevant sub-stage red but, per the orchestrator's
  fail-safe mode, **does not abort** the pipeline.

## Notes

- The legacy `FAIL_ON_VIOLATION=false` is intentional during initial
  rollout — keep it. Flip to `true` once the team has triaged baseline
  findings.
- Scans run before the artifact stages, so a missing `RESOLVED_SHA`
  (e.g. resolve-version stage failed) **should** cause scans to fall
  back to `staging` HEAD — the module accepts an empty `COMMIT_SHA`.
