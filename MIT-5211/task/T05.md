# T05 — Production hardening (NetworkPolicy + PDB + HPA)

**Status**: pending
**Repo**: `takamulai/silmarils` — append tasks to `15b-apisix-silmarils-lfi.yaml`; three new templates
**Blocks**: T06
**Blocked by**: T01 (G7 — `cloudflared` SA name + namespace), T04 (data plane Ready)

## Reason

PLAN.md commits to NetworkPolicy default-deny, PDB `minAvailable: 1`,
and HPA (CPU 70%, qa 2-4) from day one. Splitting them from T04 keeps
T04's acceptance simple ("pod Ready, /healthz 200") and layers policy
guardrails on top of a known-working data plane — easier to debug NP
denials when you know the pod is fine.

## Input

- T01 G7 result: `cloudflared` ServiceAccount name + namespace.
- PLAN.md §Production hardening.
- AKS cluster CNI mode (`kubectl -n kube-system get ds | grep -E 'azure|calico'`) — NetworkPolicy enforcement requires Azure CNI w/ NP enabled, Calico, or Cilium. If kubenet, NPs are silently no-op — escalate as a deviation.

### `templates/apisix-networkpolicy.yaml.j2`

NetworkPolicy `apisix-default-deny-plus-cloudflared`:
- `podSelector.matchLabels: { app: apisix }`.
- `policyTypes: [Ingress, Egress]`.
- Ingress:
  - From `namespaceSelector` matching the `cloudflared` namespace **and**
    `podSelector` matching `serviceAccountName: <cf-sa>` (via the
    auto-applied `kubernetes.io/service-account.name` label, or
    whatever the cloudflared deploy labels its pods with — confirm in
    T01 G7).
  - Ports 19880/TCP, 19888/TCP.
- Egress:
  - To `namespaceSelector` matching `silmarils-{{ silmarils_environment }}`
    + `podSelector` matching `app: dc-tang`, port 28888/TCP.
  - To `namespaceSelector` matching `silmarils-{{ silmarils_environment }}`
    + `podSelector` matching the in-cluster simulator labels (port 8080/TCP).
  - To `kube-system` `kube-dns` on 53/UDP+TCP for DNS resolution.

If LFI upstreams are external IPs (G8 reveals this), add an
`ipBlock`-based egress rule.

### `templates/apisix-pdb.yaml.j2`

- `spec.minAvailable: 1`.
- `selector.matchLabels: { app: apisix }`.

### `templates/apisix-hpa.yaml.j2`

- `autoscaling/v2`.
- `scaleTargetRef`: Deployment `apisix`.
- `minReplicas: {{ silmarils.apisix.autoscaling.min_replicas }}`,

## Acceptance criteria

- Re-run of the full playbook keeps pod Ready and adds NP/PDB/HPA.
- `kubectl -n apisix-silmarils get networkpolicy,pdb,hpa` all present.
- Deny test: from an unrelated pod in `silmarils-qa`, curl APISIX —
  expect connection timeout / refused.
- Allow test: from a pod with the `cloudflared` SA in its namespace
  (or temporarily annotate a curl pod to mimic), curl APISIX — expect
  200.
- HPA reports CPU load and `unknown`/`<target>` correctly (with metrics-server present).
- PDB blocks a forced drain when only 2 pods are running.

## Notes

- Verify AKS CNI before relying on policy enforcement (input bullet).
  Kubenet → silent no-op → critical security gap; escalate.
- If the `cloudflared` deploy's pod labels don't include a stable
  identifier, fall back to `namespaceSelector`-only ingress with a
  comment that the namespace itself is the trust boundary.
- HPA needs metrics-server installed in the cluster; silmarils has it
  per the monitoring playbook, but assert in preflight.
- Egress to external LFI IPs (G8) is the most likely place this
  policy will need a tweak post-deploy; keep the template scaffold
  ready for an `ipBlock` list driven from `silmarils.apisix.org_routing`.
