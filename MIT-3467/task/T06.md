Mirrors the `01-jenkins.yml` shape so SonarQube becomes another numbered playbook in the cicd ansible suite. Deploying SonarQube is a same-cluster, same-namespace concern — the playbook is thin: KV-fetch → Secrets → Helm install → SA annotate → wait → summary.

- Read `[SIBLING] cicd/ansible/playbooks/inc/fetch-and-store-secret.yaml` and `inc/credential-stores/azure_kv/get-secret.yaml` — the KV fetch helper used in the secret loop.

## Output

### `[SIBLING] cicd/ansible/playbooks/06-sonarqube.yml`

Tasks (use the same task naming style as `01-jenkins.yml`):

1. **Pre-flight + namespace**
   - Create namespace `{{ cicd_namespace }}` (idempotent).
2. **Init secret bag**
   - `set_fact: _cicd_secrets: {}`
3. **Fetch all `cicd_secret_names` from KV**
   - Loop `cicd_secret_names | dict2items` → `include_tasks: inc/fetch-and-store-secret.yaml`. Harmless re-fetch of Jenkins keys (per design — F11).
4. **Assert SonarQube keys**
   - `assert:` that all six KV keys (`sonarqube_admin_password`, `sonarqube_current_admin_password`, `sonarqube_db_password`, `sonarqube_monitoring_passcode`, `sonarqube_saml_cert`, `sonarqube_token`) are present in `_cicd_secrets`.
5. **Build `secret.properties` content** (in-memory)
   - `set_fact:`
     ```yaml
     _sonarqube_secret_properties: |
       sonar.auth.saml.certificate.secured={{ _cicd_secrets[cicd_secret_names.sonarqube_saml_cert] }}
     ```
6. **Create K8s Secret `sonarqube-credentials`** in namespace `{{ cicd_namespace }}` via the `kubernetes.core.k8s` module (or the `oc-apply-idempotent` pattern used elsewhere) with `stringData`:
   - `admin-password`: target admin pwd
   - `current-admin-password`: current admin pwd (literal `admin` first time — F10)
   - `db-password`: bundled Postgres password
   - `monitoring-passcode`: monitoring endpoint passcode
   - `saml-certificate`: SAML cert PEM body (also lands inside `secret.properties` for the chart)
   - `secret.properties`: contents of `_sonarqube_secret_properties` from step 5
7. **Helm repo add + update**
   ```
   helm repo add sonarqube https://SonarSource.github.io/helm-chart-sonarqube
   helm repo update sonarqube
   ```
8. **Render values file with SAML app id** — `template:` module renders T05's `values.yaml` substituting `__SAML_APP_ID__` from `sonarqube_saml_application_id` group_var, output to `/tmp/values-sonarqube-rendered.yaml`.
9. **Helm upgrade --install**
   ```
   helm upgrade --install sonarqube sonarqube/sonarqube \
     --namespace {{ cicd_namespace }} \
     --version {{ sonarqube_helm_version }} \
     -f /tmp/values-sonarqube-rendered.yaml \
     --wait --timeout 15m0s
   ```
10. **Patch SA annotation** — `kubectl annotate serviceaccount sonarqube -n {{ cicd_namespace }} --overwrite azure.workload.identity/client-id={{ sonarqube_workload_identity_client_id }}` (F13).
11. **Wait for StatefulSet ready** — retry up to 60× with 10s delay until `kubectl get sts sonarqube-sonarqube -n {{ cicd_namespace }} -o jsonpath='{.status.readyReplicas}'` is `>= 1` (cold start ~3 min).
12. **Summary**
    - `debug:` echoing `{{ sonarqube_url }}` and rendered status.
    - Optional: `file: state=absent path=/tmp/values-sonarqube-rendered.yaml` to clean up.

## Notes

- The KV fetch helper requires the operator's CLI to be authenticated to the lower subscription (`az login` against the right tenant). Document this prereq in T08's runbook.
- The SAML cert PEM goes into the Secret in **two** places: as a top-level `saml-certificate` key (for any consumer that wants it raw) and inside `secret.properties` (for the chart's `sonarSecretProperties` mechanism — F12). One source of truth (KV), two destinations.
- Don't add a destroy playbook for this rollout — Helm uninstall + namespace cleanup is sufficient if needed manually. A formal destroy is a follow-up.
- Tag tasks consistently (e.g. `[sonarqube, deploy]` / `[sonarqube, secrets]`) so operators can `--tags secrets` to re-fetch alone.
