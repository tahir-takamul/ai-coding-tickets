## Reason

T16's GHA → Jenkins → Sonar pipeline has one accepted limitation (FINDING §F28): concurrent PR scans into the same base branch race on the shared Sonar project key (`mithril-${component}-${base}`). Sonar Community Build has no native per-branch / per-PR analysis, so each PR scan overwrites the previous. Result: the pass/fail Check is still correct (driven by Jenkins via `qualitygate.wait=true`), but the enrichment metrics + inline comments in the PR Check + summary comment may reflect *another* PR's data.

T17 adopts the third-party `sonarqube-community-branch-plugin` (<https://github.com/mc1arke/sonarqube-community-branch-plugin>) to give Community Build real branch + PR analysis. Closes F28; gives correct "new code" semantics scoped per-PR vs. target; collapses the 6 long-lived projects to 3.

### Step 3 — Migrate Jenkinsfile scanner invocations

`iac/eng-infra/cicd/pipelines/modules/sonar-scan.Jenkinsfile`:

```groovy
// OLD: per-branch project keys
def projectKey = "mithril-${params.COMPONENT}-${params.BRANCH}"

// NEW: single per-component project key; branch/PR scope via scanner props
def projectKey = "mithril-${params.COMPONENT}"

// Then per-component sh blocks:
// For push scans / direct main scans:
sh "./gradlew sonar -Dsonar.projectKey=${projectKey} -Dsonar.branch.name=${params.BRANCH} ..."

// For PR scans (sonar-pr wrapper passes PR_NUMBER + PR_BRANCH_HEAD + PR_BRANCH_BASE):
sh "./gradlew sonar -Dsonar.projectKey=${projectKey} -Dsonar.pullrequest.key=${params.PR_NUMBER} -Dsonar.pullrequest.branch=${params.PR_BRANCH_HEAD} -Dsonar.pullrequest.base=${params.PR_BRANCH_BASE} ..."
```

`sonar-pr.Jenkinsfile` needs to forward both `PR_BRANCH_HEAD` and `PR_BRANCH_BASE` (currently only forwards `PR_BRANCH` which we resolved to base). Update the JCasC `genericVariables` to map both, and the GHA payload to send both.

### Step 4 — Update composite action

`.github/actions/sonar-pr-report/action.yml`:

- Drop the `pr-branch` → project-key-suffix logic.
- Replace `PROJECT_KEY="${prefix}-${component}-${pr_branch}"` with `PROJECT_KEY="${prefix}-${component}"`.
- API calls switch from per-project to per-PR scope:
  - `GET /api/measures/component?component=${PROJECT_KEY}&pullRequest=${PR_NUMBER}&metricKeys=...`
  - `GET /api/qualitygates/project_status?projectKey=${PROJECT_KEY}&pullRequest=${PR_NUMBER}`
  - `GET /api/issues/search?componentKeys=${PROJECT_KEY}&pullRequest=${PR_NUMBER}&resolved=false`
- Sonar dashboard link → `${SONAR_URL}/dashboard?id=${PROJECT_KEY}&pullRequest=${PR_NUMBER}` (plugin renders the PR view).

Optionally retire our composite action's PR decoration (summary comment + inline comments) in favor of the plugin's native GitHub decoration. Trade-off: less code to maintain vs. less control over format. Decide at implementation time after seeing what the plugin's decoration looks like in practice.

## Notes

- **Risk**: Plugin is third-party, not SonarSource-supported. Upgrade ergonomics: when SonarQube version bumps, also bump the plugin version. Maintainer's track record is good (releases for every SQ release since 7.x).
- **Decoration overlap**: if the plugin's PR decoration is left enabled alongside our composite action's, the PR will get two sets of comments. Disable one side via plugin settings (`sonar.pullrequest.provider`) or remove the composite action's posting steps. Decide at implementation.
- **Revert**: `helm upgrade` back to the values from T08 (without `plugins.install` + `javaagent` opts + init container). Sonar PVC retains all data. Projects revert to the per-branch model on the next scan.
