## Reason

Jenkins agent pods run from `mithrilacr.azurecr.io/cicd-agent:<tag>`. To run `sonar-scanner` in a pipeline (smoke test, T11; future per-module scans), the binary must be on `PATH` in the agent.

Per FINDING ¬ßF17, instead of building a dedicated scanner image we append the scanner CLI to the existing `cicd-agent` image. Sonar Scanner CLI 6.x bundles its own JRE, so no JDK is required for the scanner itself.

## Output

### Append to `[SIBLING] cicd/docker/cicd-agent/Dockerfile`

Insert after the existing tool-install RUN blocks:

```dockerfile
ARG SONAR_SCANNER_VERSION=6.2.1.4610
RUN curl -sSL "https://binaries.sonarsource.com/Distribution/sonar-scanner-cli/sonar-scanner-cli-${SONAR_SCANNER_VERSION}-linux-x64.zip" -o /tmp/sonar.zip && \
    unzip -q /tmp/sonar.zip -d /opt && \
    mv /opt/sonar-scanner-${SONAR_SCANNER_VERSION}-linux-x64 /opt/sonar-scanner && \
    ln -s /opt/sonar-scanner/bin/sonar-scanner /usr/local/bin/sonar-scanner && \
    rm /tmp/sonar.zip && \
    sonar-scanner --version
```

## Notes

- `sonar-scanner --version` smoke at the end of the RUN block ensures the install succeeded at build time, not just at first runtime use.
- If the base image is **not** `alpine/k8s:1.32.11` and instead lacks `unzip`, prepend `apt-get install -y unzip` (or `apk add --no-cache unzip`) ‚Äî verify before committing.
- Don't pin the `SONAR_SCANNER_VERSION` to anything older than 6.x ‚Äî older versions bundle Java 11 which is past EOL for the scanner ecosystem.
- ~~Do NOT install a system JDK into `cicd-agent` purely for the scanner~~ ‚Äî **superseded 2026-05-08 by FINDING ¬ßF19b**: backend's `./gradlew test` is a real JDK consumer, not "the scanner alone". The Dockerfile now installs full `openjdk25` (replacing `openjdk17-jre-headless`); the same JVM serves both the scanner CLI and `./gradlew`. The original spirit ("don't add a JDK gratuitously") still applies ‚Äî adding it for backend builds is a different, justified case.
- The JDK 25 image is published as `cicd-agent:sonar-jdk25-<sha>` + sliding `cicd-agent:sonar-jdk25`. **`:latest` is intentionally NOT retagged** ‚Äî the five non-sonar orchestrator pipelines (`deploy*`, `demo-orchestrator`, `qa-orchestrator`, `staging-orchestrator`) keep running on the JRE-17 image. Only `sonar-scan.Jenkinsfile` pins to `:sonar-jdk25`. See FINDING ¬ßF19b for the regression-risk reasoning.
- Gradle toolchain auto-download (`foojay-resolver-convention`) was considered as an alternative and rejected: backend's `settings.gradle` doesn't apply the plugin today, and downloading a JDK on every fresh agent is worse than baking it into the image.
