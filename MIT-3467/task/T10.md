# T10 — First-login hardening: rotate admin pwd, generate Jenkins token, push to KV

**Status**: pending
**Repo**: mixed — Sonar UI (browser) + KV (CLI) + `[SIBLING]` Ansible re-run
**Blocks**: T11 (smoke pipeline needs the token)
**Blocked by**: T08, T09

## Reason

After the playbook deploys SonarQube, the chart has rotated admin from `admin` to the value in `cicd-sonarqube-admin-password`. We still need to:
1. Verify SAML login works for a real user (DevOps Admin group member).
2. Generate a **user token** for Jenkins to authenticate scans (project analysis tokens work too, but a single user token simpler for this rollout's smoke scope).
3. Push the token to KV as `cicd-sonarqube-token`.
4. Re-run `01-jenkins.yml` so Jenkins's secret bag picks up `cicd-sonarqube-token` and surfaces it as a Jenkins credential consumable by pipelines.

## Input

- T08 done; SonarQube reachable internally and at `https://sonarqube.takamul.cc`.
- T09 done; CF Access lets browser users in (with Entra SSO challenge from CF first).
- Operator has access to `mithril-shared-kv` (`Set` perm, not just Get).
- Operator's account is a member of `Takamul DevOps Admin` (Entra group `0309fec6-57e6-4446-bf4f-af084823d03a`).

## Output

### Step 1 — confirm admin password rotated

Log in to `https://sonarqube.takamul.cc` as **`admin`** with the password in `cicd-sonarqube-admin-password` (target). If login fails:
- Try `admin/admin` — chart hasn't rotated. Update `cicd-sonarqube-current-admin-password` to current value, re-run `06-sonarqube.yml`.

If login succeeds, rotation worked. (Optional: rotate again now and update both KV secrets — defensive.)

### Step 2 — verify SAML SSO

Log out. On the Sonar login page, click **"Log in with Azure Entra ID"**. CF Access has already authenticated the browser; this kicks off Sonar's own SAML flow. Confirm:
- Redirect to `login.microsoftonline.com/<tenant>/saml2`.
- Returns to Sonar logged-in as the operator's Entra identity.
- Operator has admin rights (because DevOps Admin group is mapped — T02).

If SAML fails: check Sonar logs (`kubectl -n cicd logs sts/sonarqube-sonarqube`) for cert / claim-mapping errors. Most common issue is the cert PEM in KV still has BEGIN/END headers (T03 should have stripped them).

### Step 3 — generate user token for Jenkins

Sonar UI → **My Account** (top-right avatar) → **Security** → **Generate Tokens**:
- Name: `jenkins-scanner`
- Type: **User Token** (not Project Analysis Token — keeps the smoke test simple; can switch to per-project tokens in the per-module follow-up).
- Expires: 1 year.
- Click **Generate** and copy the token value (shown once).

### Step 4 — push token to KV

```
az keyvault secret set --vault-name mithril-shared-kv \
  --name cicd-sonarqube-token \
  --value '<token from step 3>'
```

This overwrites the placeholder set in T03.

### Step 5 — re-run `01-jenkins.yml`

From `[SIBLING] cicd/ansible/`:

```
ansible-playbook -i inventory/<env>/hosts playbooks/01-jenkins.yml --tags secrets
```

`--tags secrets` (or whatever the Jenkins playbook uses to scope to the secret-fetch step) avoids re-doing the full Jenkins reinstall. If the Jenkins playbook doesn't tag its secrets stage, run the whole playbook — Helm should be a no-op on subsequent runs.

After this, the `cicd-sonarqube-token` lands in Jenkins's credential store as a Jenkins **Secret Text** credential (the exact credential ID matches the convention used by Jenkins's other KV-backed creds — verify via Jenkins UI → Manage Jenkins → Credentials).

## Acceptance criteria

- Log in to SonarQube as `admin` with the KV-stored target password — succeeds.
- SAML SSO flow completes; Entra-authenticated user lands logged-in as a Sonar admin.
- `az keyvault secret show -n cicd-sonarqube-token --vault-name mithril-shared-kv --query value -o tsv` returns the new token (not `placeholder`).
- Jenkins Credentials page lists a credential whose value (verifiable via mask in build log) matches the Sonar token.

## Notes

- **One token, not three**. The pasted plan's smoke test uses `projectKey=smoke-test` — a single placeholder. When per-module scans land (follow-up), generate three project-scoped tokens (`jenkins-scan-backend`, etc.) and split into `cicd-sonarqube-token-{backend,portal,wallet}`.
- Set a calendar reminder to rotate this token before the 1-year expiry. Expired token = silent scan failures.
- Do NOT paste the token value into any task file or commit.
- If the operator is **not** a DevOps Admin group member, SAML login will succeed but the user will land without admin rights. Either join the group or temporarily add another group to the SP role assignment in T02 — the latter is a TF change, not a Sonar change.
