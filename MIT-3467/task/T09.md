# T09 — Cloudflare tunnel + Access app/policy for `sonarqube.takamul.cc`

**Status**: pending
**Repo**: `[CF] cloudflare/` (separate platform-team repo — not on the local machine)
**Blocks**: T10 (browser SSO needs the URL), T11
**Blocked by**: T08

## Reason

External access to SonarQube (browsers + future GHA Web API callers) goes through the existing Cloudflare tunnel `eng-infra-azure`, the same tunnel that fronts `jenkins.takamul.cc`. A new ingress rule is appended; the Access app + policy auto-generates via the `for_each = local.cicd_access_apps` pattern.

See FINDING §F14, §F15.

## Input

- Read `[CF] cloudflare/terraform.tfvars` lines 241-248 — current `cicd_ingress_rules` array, contains the Jenkins entry. Mirror its shape.
- Read `[CF] cloudflare/main.tf` lines 406-440 — the `for_each = local.cicd_access_apps` block that auto-generates Access app + policy from each ingress rule. Confirm no manual Access-app addition is needed.
- T08 deploy completed — internal Service `sonarqube-sonarqube.cicd.svc:9000` is responding `UP`.

## Output

### Update `[CF] cloudflare/terraform.tfvars`

Append a SonarQube entry to `cicd_ingress_rules`:

```hcl
cicd_ingress_rules = [
  {
    hostname = "jenkins.takamul.cc"
    service  = "http://jenkins.cicd.svc:8080"
  },
  {
    hostname                 = "sonarqube.takamul.cc"
    service                  = "http://sonarqube-sonarqube.cicd.svc:9000"
    options_preflight_bypass = false
  },
]
```

The `for_each` Access-app loop in `main.tf:406-440` picks up the new entry automatically — no `main.tf` edits needed.

### Apply

```
cd cloudflare/
terraform plan
terraform apply
```

Coordinate with the platform/CF repo owner; this is cross-team.

## Acceptance criteria

- `curl -I https://sonarqube.takamul.cc` (no headers) returns 302 to a Cloudflare Access challenge URL (proves CF Access is enforcing).
- `curl -H "CF-Access-Client-Id: …" -H "CF-Access-Client-Secret: …" https://sonarqube.takamul.cc/api/system/status` (with valid service-token headers) returns `{"status":"UP"}` (proves the tunnel reaches the in-cluster service and the Access policy allows the service token).
- DNS: `dig +short sonarqube.takamul.cc` returns the Cloudflare-managed CNAME target.

## Notes

- The same SSO policy used by `jenkins.takamul.cc` applies — Entra-tied. Browser users go through CF Access first, then through Sonar's own SAML (two layers, intentional, mirrors Jenkins).
- The Access service token used by Jenkins / GHA today is added to the new app's allowed service tokens automatically by the `for_each` block (verify by reading the `main.tf` block — if it doesn't auto-add, an explicit allow-list entry is required).
- DO NOT expose SonarQube without CF Access — the bundled Postgres credentials are only as strong as the network boundary if Access is bypassed. Without Access, anyone on the internet hits the SonarQube login page; with Access, they hit Cloudflare's challenge first.
- DO NOT add an internal hostname (`sonarqube.apps.<cluster>.takamul.cc`) — Jenkins agents in the same AKS cluster reach Sonar via in-cluster Service DNS (F06), not via any tunnel or Route.
